# ICR Project Documentation

## Project Overview
This is a single-instance Rails application designed to run for a single company. There is no multi-tenancy support, and company details are managed through a singleton settings model.

### Model Organization
- Employee-related models live under the `Employee::` namespace and use explicit `class_name` in associations (required for the namespace to resolve). Serializers mirror the same namespace.

## Conventions
### Architecture
- Single-tenant application
- Company settings use singleton pattern

### Controller Concerns
- Config used in instance methods (e.g. sortable fields) must use `class_attribute` so each including class gets its own config. Module-level instance variables set in `included` are class ivars and are not visible in instance methods.

### Settings Controllers
- Singleton settings share one update endpoint across multiple Inertia pages. Use `redirect_back_or_to settings_path` so both success and validation errors return to the originating page without losing Inertia error handling.

### Development Guidelines
- No multi-tenancy support
- Changes must be atomic and focused
- When checking HTTP response status, use exact status equality (e.g. 201), not range checks.
- TypeScript: strict types, prefer existing types and generics over inline types or assertions; only annotate return types when intent is ambiguous or for public API.
- Nested ternaries are disallowed; use `if`/`else` or early returns.

### Permission System
- Groups define permissions with effects (allow/deny) and conditions. Assignments link groups to users or job titles (optional time bounds). Resolution uses precedence and conditions.
- Precedence: higher overrides lower. At each level, deny wins over allow.
- Conditions: owner-only, time-restricted, or custom. Check with permission?(resource, action, context).

### Payroll Settings
- Period types: weekly, bi-weekly, semi-monthly, monthly; each with configurable start day(s) or dates. Monthly can use last day of month.
- Overtime: configurable weekly/daily thresholds; exemption rules by shift category, time range, duration.
- Pay rates: historical records with effective dates; hourly and salary; currency.

## Technical Decisions
### Authentication & Authorization
- Devise, Pundit, Rolify; authorization at API and UI. Custom permission system adds fine-grained rules with precedence, conditions (time-based, owner-only), and polymorphic assignments (users, job titles).

### Document Management
- Active Storage; versioning for important documents; Pundit for access.

### Calendar System
- Multiple view strategies; timezones must be handled correctly; recurring events follow RFC 5545 (iCalendar).
- One display-setting drives both month and time-grid views (month strategies vs overlap for day/week), so per-strategy behavior can be added without changing the public API.
- Gotcha: The popover closes on click-outside. Date/time pickers or other inputs inside the popover must not use a portal, or the popover will close when the user selects a value.
- Client schedule: Events come from React Query; mutations must invalidate the schedule query so the calendar refetches. If invalidation is skipped, the UI can show stale or duplicate events.
- Client schedule uniqueness: Participantable's calendar_events is has_many through event_participants; the join can return the same event multiple times. The association uses a distinct scope so the API returns each event once.

### Timesheet System
- Payroll index: employees only; historical view deferred.
- Employee payroll review: per-employee view for the current pay period; authorization follows timesheet show.
- PayPeriod: one per (starts_at, ends_at); no overlapping periods; derived from settings.
- Timesheet: pre-submittal; tracks approval; belongs to PayPeriod and Employee; has many Shifts.
- Shift create: if within current pay period, find or create that PayPeriod and a Timesheet for the employee; if the PayPeriod was just created, backfill timesheets for all shifts in that range.
- PayPeriod create: backfill timesheets from shifts whose event overlaps the period (one Timesheet per employee).
- Payroll index hours: Payroll::EmployeePeriodHours computes per-employee regular_hours and ot_hours for a pay period. Uses Setting overtime_weekly_hours and overtime_daily_hours; splits shifts by day (including overnight); applies active OvertimeExemption rules to exclude exempt shift hours from OT; weekly OT and daily OT are summed. Overtime exemption category criteria use Shift.category (Shift's own category_id), not calendar event category.
- Payroll approval UI: Approve actions (index row menu, footer bulk approve, employee review Approve button) are visible only during the approval window (current date within 3 days before period end through period end).

### Frontend Components
#### UI Framework
- Mantine v8+; use current API from docs.
- Gotcha: Mantine date/time inputs expect strings; shared inputs accept Date or string and normalize to ISO string before passing in.

#### Spotlight (command palette)
- Single instance in the shared Providers tree; actions are permission-gated and mirror sidebar navigation.

#### Icons
- Single export from components/Icons; icons renamed for semantic use. IconProvider sets default size and class.

#### Component Organization
- Frontend: components, features, layouts, lib, pages, queries, types under app/frontend.
- React 19: ref is passed as a normal prop; components that need ref support accept ref in their props interface (ref?: React.Ref&lt;HTMLElement&gt;) and pass it to the underlying DOM or child. forwardRef is not used.

#### Links and navigation
- Use the Link component with `as="button"` and `buttonProps` for button-styled navigation (not `Button component={Link}`); Link with as="button" renders an anchor styled as a button and uses Inertia routing.
- Use the Routes object from @/lib for all hrefs; do not build URLs manually (e.g. use Routes.payrollEmployeeReview(employeeId), not `/payroll/employees/${id}`).

#### Formatters
- All formatting lives in formatters.ts; do not define formatters inline. Add new ones there and use via formatter / formatter.datetime / formatter.number from @/lib.


#### UI & Styling
- Mantine; Linaria with .css.ts; clsx for class composition; theme supports dark/light.

#### Form Implementation
- use-inertia-form v4.6.0; Form wraps Inertia + Mantine; nested data, Rails-style attributes; custom inputs via the form hook; strict typing.
- Gotcha: To submit via a custom path (e.g. React Query), use the optional submit-with callback; when set, default submit is skipped and the normal destination is optional.

#### State Management
- Server state: Inertia.js
- Client-only state: zustand
- Form state: use-inertia-form

#### Type System
- Strict TypeScript; prefer existing types and generics; no any/unknown or casting.
- Ruby constants can be exported to TS via rails types:generate; add a subtask in types.rake so it stays in sync.
- moduleResolution "bundler" is required for subpath exports (e.g. tiptap menus).

#### Internationalization (i18n)
- react-i18next; locales from Rails (i18n-js). Model-specific frontend strings in config/locales/[model]/frontend.en.yml; rest in frontend.en.yml. Keys: views.[feature].[page].[key]. Controller notices use shared templates under templates.controllers.notices with %{model}; CRUD notices (created/updated/destroyed/status_updated) use these.

#### Testing
- Unit: Vitest + RTL; E2E: Playwright (no mocks). Tests mirror app/frontend under app/frontend/tests.
- Gotcha: For assertions on translated strings, run validation and assertions inside I18n.with_locale(:en) so the expected message is stable.
- Gotcha: Forms must use remember={ false } or the form library can persist state across unmount/remount and tests see stale values.
- Gotcha: For form-context tests, use Form from use-inertia-form (not the app Form wrapper). For Mantine, wrap with MantineProvider, stub matchMedia in beforeEach, and wrap Grid.Col in Grid.
- Do not test literal constants from the source; test behavior, branches, and contracts.

#### Serializers
- Smoke spec discovers serializers, infers model from name, renders each. Slug-identified models need persisted records so the smoke spec creates them. Gotcha: Identification uses the id_type column; serializers must use :id_type, not :type.

#### Lib coverage
- spec/lib covers lib code; generator specs live under spec/lib/generators. Gotcha: In specs that register at-exit blocks with doubles, stub the class's at_exit so real at-exit is not registered.

#### Performance
- Code splitting, lazy loading, memoization, minimal global state.

#### TipTap Integration
- TipTap v3.14.0; Mantine integration via @mantine/tiptap. Gotcha: Menu components (BubbleMenu, FloatingMenu) must be imported from the /menus subpath, not the main package.

## Payroll Periods

### Semi-Monthly
- With “last day of month” setting: first period 1st-14th, second 15th-last day. Otherwise two configurable dates; each period ends the day before the next starts.

### Bi-Weekly
- Fixed reference date for boundaries; 14-day periods; configurable start day of week.

### Weekly
- Configurable start day of week; 7-day periods.

### Monthly
- Configurable start date or “last day of month.” When last day: period is last day of prior month through second-to-last day of current month. Otherwise ends day before next period's start.
