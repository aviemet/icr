# ICR Project Documentation

## Project Overview
This is a single-instance Rails application designed to run for a single company. There is no multi-tenancy support, and company details are managed through a singleton settings model.

### Model Organization
- Employee-related models are namespaced under `Employee::`
  - `Employee::JobTitle`: Manages job titles and their associations
  - Models use explicit class names in associations (e.g., `class_name: "Employee::JobTitle"`)
  - Serializers follow the same namespace pattern (e.g., `Employee::JobTitleSerializer`)

## Conventions
### Architecture
- Single-tenant application
- Company settings use singleton pattern

### Controller Concerns
- Concerns that store config used in instance methods (e.g. Searchableâ€™s sortable fields, advanced search methods) use `class_attribute` so includers get per-class config visible to instances. Avoid module-level instance variables in `included` blocks; those are class ivars and are not available in instance methods.

### Settings Controllers
- Singleton `Setting` updates share the `/settings` update endpoint across multiple Inertia pages (e.g. General, Calendar); controllers use `redirect_back_or_to settings_path` so successful updates and validation failures return the user to the originating settings page while preserving Inertia error handling.

### Development Guidelines
- No multi-tenancy support
- Changes must be atomic and focused
- When checking HTTP response status, use explicit status code equality only (e.g. `res.status === 201`). Never use range or inequality checks (e.g. `res.status >= 200`, `res.status < 300`).
- TypeScript is used for all JavaScript code
  - Start with most restrictive type definitions
  - Avoid `any`, `unknown`, and type casting with `as`
  - Prefer existing type definitions and interfaces
  - Use generics, Records, and Mapped types before inline types
  - Type assertions are a last resort
  - Do not explicitly annotate function return types when TypeScript can infer them; only add a return type when the intent is ambiguous or part of a public API.
- Nested ternaries are not allowed; use normal `if`/`else if`/`else` (or early returns) instead.

### Permission System
The permission system provides fine-grained access control with support for complex conditions:

#### Components
- **Permission Groups**: Define sets of permissions with effects (allow/deny) and conditions
- **Permission Assignments**: Link permission groups to users or job titles with optional time restrictions
- **Permission Resolution**: Evaluates permissions based on precedence levels and conditions

#### Key Features
- **Precedence Levels**: Higher precedence permissions override lower ones
- **Explicit Allow/Deny**: At each precedence level, deny takes priority over allow
- **Conditional Access**: Supports conditions like:
  - Owner-only access (user must own the record)
  - Time-restricted access (only valid during specific time windows)
  - Custom conditions (extensible for specific needs)

#### Usage
```ruby
# Creating a permission group
permission_group = Permission::Group.create!(
  name: "Time-restricted View",
  description: "Allow viewing during business hours",
  permissions: [{
    resource: "clients",
    action: "view",
    effect: "allow",
    conditions: {
      time_restricted: {
        start_time: "09:00",
        end_time: "17:00"
      }
    }
  }]
)

# Assigning permissions
Permission::Assignment.create!(
  permissionable: user,
  group: permission_group,
  starts_at: Time.current
)

# Checking permissions
user.permission?("clients", "view", context)
```

### Payroll Settings
The payroll settings system manages pay periods and overtime calculations:

#### Pay Period Configuration
- **Period Types**:
  - Weekly
  - Bi-weekly
  - Semi-monthly
  - Monthly
- **Period Settings**:
  - Weekly/Bi-weekly: Configurable day of week (Monday-Sunday)
  - Monthly: Configurable date (1st, 5th, 15th, 20th, or last day)
  - Semi-monthly: Two configurable dates per month

#### Overtime Rules
- **Standard Thresholds**:
  - Weekly hours (default: 40)
  - Daily hours (default: 8)
- **Exemption System**:
  - Configurable rules for overtime-exempt shifts
  - Criteria based on:
    - Shift category (Sleep, Standby, Regular)
    - Start/end times
    - Shift duration

#### Pay Rate Management
- Tracks employee pay rates over time
- Supports both hourly and salary periods
- Includes currency information
- Maintains historical records with effective dates

### Calendar System
- Calendar must support multiple view strategies
- Events must handle timezone conversions properly
- Recurring events must follow RFC 5545 (iCalendar) specification
- Calendar uses a unified `onClick` handler that accepts a `CalendarClickTarget` discriminated union
- Display strategy selection (`calendar_layout_style`) now drives both month and time-grid (day/week) views:
  - Month view maps `split`, `stack`, `span` to dedicated month strategies
  - Day/Week time-grid views alias these names to their overlap strategy and receive the chosen strategy via `displayStrategy` so future per-strategy time-grid behaviors can be wired without changing the public API
- Month view event rows use `MonthEvent` + `CalendarEvent` composition:
  - `MonthEvent` applies layout sizing and indicator/continuation visuals (dots and triangles) via `MonthView.css` and `Event.css`
  - `CalendarEvent` wraps children in a `div > span` structure that owns shared event typography and click handling
  - Target types: `event` and `background`
  - Calendar component accepts `onClick?: (target: CalendarClickTarget) => boolean | void` prop
  - Popover content configured via `popoverContent` prop object with `event` and `background` keys
  - Default popover behavior is used unless `onClick` returns `true`
  - EventPopover renamed to CalendarPopover to reflect generic nature
  - `CalendarPopover` receives `popoverRef: React.RefObject<HTMLDivElement>` as a prop (not `forwardRef`) to avoid type assertions and keep typing strict
  - Draft events: Calendar maintains an internal draft-events store and merges draft events into `events` before view strategies run. Draft events can be created/updated/removed via `useCalendarContext` methods (`upsertDraftEvent`, `patchDraftEvent`, `removeDraftEvent`).
  - Inputs rendered inside the calendar popover should avoid portal-based dropdowns/popovers (e.g. pass `popoverProps={{ withinPortal: false }}` to date/time pickers) so calendar click-outside logic does not close the popover on selection

### Timesheet System
- **Payroll index** (`/payroll`): List of employees only; historical view (e.g. `/payroll/historical`) deferred.
- **PayPeriod**: Snapshot of an actual pay period (derived from settings via `Payroll::Period.period_dates`); unique on `(starts_at, ends_at)`; no overlapping periods.
- **Timesheet**: Created before submittal; tracks approval status; belongs to PayPeriod and Employee; has many Shifts.
- **Shift â†’ Timesheet/PayPeriod**: On create, if shift falls within the active pay period (current date's period): find or create PayPeriod for that span; if PayPeriod was newly created, its `after_create` backfills timesheets from all shifts in range; otherwise find or create Timesheet for that employee and PayPeriod and set `shift.timesheet_id`.
- **PayPeriod â†’ Timesheets**: On create, `backfill_timesheets_from_shifts`: find shifts whose calendar_event overlaps the period, group by employee, create one Timesheet per employee, associate those shifts with the Timesheet.
- Timesheets will be generated per pay period; audit trail and multi-step approval workflow as before.

- Unit: Vitest + React Testing Library
- E2E: Playwright (no mocks, uses test environment)
- Test files mirror source structure in `/app/frontend/tests`
- When a spec asserts on a translated validation/error message string, run the validation and assertions inside `I18n.with_locale(:en)` so the expected message is stable regardless of test order or default locale
- Run tests when you make changes.
  - Rails tests are in the rspec folder using standard naming Conventions
  - JS/TS tests are in app/frontend/tests and follow the same folder structure as the files they're testing
  - If no tests for the feature being worked on exist, create them.
- Do not write tests that only re-assert literal constants from the source (e.g. asserting that an exported constant object has the same keys/values as in the file). Such tests duplicate the implementation, add maintenance without catching real bugs, and have no independent notion of correctness. Prefer testing behavior, branches, and contracts; skip tests for pure static data exports
- React component tests (Vitest + RTL): Use `render()` from `@testing-library/react`; assert on behavior with `screen` and `userEvent`. ## Payroll Periods

### Semi-Monthly Periods
- When configured with last day of month setting (`payroll_period_date_2 = -1`):
  - First period: 1st to 14th of the month
  - Second period: 15th to last day of month
- With regular settings:
  - Periods are defined by `payroll_period_date` and `payroll_period_date_2`
  - First period starts on `payroll_period_date`
  - Second period starts on `payroll_period_date_2`
  - Each period ends the day before the next period starts

### Bi-Weekly Periods
- Uses a reference date of January 3rd, 2024 for calculating period boundaries
- Periods are two weeks long (14 days)
- Start day of week is configurable via `payroll_period_day`

### Weekly Periods
- Start day of week is configurable via `payroll_period_day`
- Periods run for 7 days from the start day

### Monthly Periods
- Can be configured with a specific start date or last day of month
- When using last day of month (`payroll_period_date = -1`):
  - Period runs from last day of previous month to second-to-last day of current month
- With regular settings:
  - Period starts on `payroll_period_date`
  - Ends on day before start date of next month
